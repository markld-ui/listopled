# Assumptions

## Принятые допущения

1. Фактический главный файл проекта — `MASTER_PROMPT_LISTOPLED.md`, хотя внутри мастер-файла местами упоминается `MASTER_PROMPT.md`.
2. Текущая итерация Фазы 1 ограничена документацией, `AGENTS.md` и changelog; backend/frontend код, Docker Compose, nginx и `.env.example` будут создаваться после готовности документации и отдельного подтверждения.
3. До появления финальных Figma-материалов используются placeholder palette из мастер-файла и материалы из `static/photos`.
4. Полный список форм листьев может быть дополнен после переноса данных с текущего Tilda-сайта.
5. Production storage пока не выбран окончательно: документация допускает Local или MinIO через `IFileStorageService`.
6. Юридически финальный текст privacy policy должен быть согласован отдельно перед production.
7. В мастер-файле есть расхождение между примером BFF routes с `leads` и основным API с `inquiries`; точное имя BFF routes требует подтверждения перед реализацией.
8. Фаза 1 до пользовательского ревью ограничена документацией и безопасным каркасом; рабочие реализации калькулятора, авторизации, админки, аналитики, обращений и публичного продукта не создаются.
9. Backend skeleton создается без внешних NuGet-пакетов, чтобы `dotnet restore/build` мог пройти без сетевой загрузки; EF Core, MediatR, AutoMapper и другие пакеты будут добавляться отдельными задачами с обоснованием зависимостей.
10. Frontend skeleton содержит Nuxt package manifest, но установка npm-зависимостей зависит от доступа к npm registry.
11. `npm install` для frontend skeleton сообщил о 6 high severity vulnerabilities в транзитивных зависимостях; `npm audit fix` не запускался, чтобы не менять dependency graph без отдельного решения.
12. `static/photos` считается локальным приватным хранилищем брендовых материалов. Реальные фото, логотипы, Figma-материалы, исходники и дизайн-референсы не коммитятся и не отправляются в Docker build context; для публичного использования изображения нужно отдельно отобрать, оптимизировать и перенести в публичную директорию frontend или загрузить в MinIO/S3.

## Вопросы перед реализацией

- Какие реальные контакты использовать для WhatsApp, Telegram, VK, email и телефона?
- Начинать production storage с Local или включать MinIO с первого релиза?
- Нужна ли email-нотификация мастеру уже в MVP?
- Какой домен и схема HTTPS будут использоваться?
- Нужно ли переносить все данные с текущего Tilda-сайта вручную до seed data?
- Утвердить `inquiries` как единый термин для BFF routes или оставить часть BFF routes с `leads`?

## Критичные решения

На момент создания документации критичных решений без утверждения пользователя не принято.
