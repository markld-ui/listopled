# AGENTS.md — LISTOPLED

Правила работы Codex в проекте LISTOPLED.

Этот файл обязателен к прочтению перед любой задачей. Он дополняет `MASTER_PROMPT_LISTOPLED.md`, но не заменяет его.

---

## 1. Главный источник истины

Перед началом любой работы Codex обязан прочитать:

1. `MASTER_PROMPT_LISTOPLED.md`
2. актуальные файлы из `doc/`
3. текущую задачу пользователя

Если задача противоречит `MASTER_PROMPT_LISTOPLED.md` или документации в `doc/`, Codex обязан остановиться, описать противоречие и предложить изменение документации.

Нельзя писать код, если непонятно, как задача соотносится с мастер-промтом.

---

## 2. Документация прежде кода

Любое значимое изменение должно быть отражено в документации.

Правила:

* изменился API — обновить `doc/API.md`;
* изменилась доменная модель — обновить `doc/DOMAIN.md`;
* изменилась схема БД — обновить `doc/DATABASE_SCHEMA.md`, если файл существует в проекте;
* изменилась архитектура — обновить `doc/ARCHITECTURE.md`;
* изменилась безопасность — обновить `doc/SECURITY_AND_PRIVACY.md`, если файл существует в проекте;
* изменился деплой — обновить `doc/DEPLOYMENT.md`;
* появилось допущение — обновить `doc/ASSUMPTIONS.md`, если файл существует в проекте;
* значимое изменение выполнено — обновить `doc/CHANGELOG.md` или `CHANGELOG.md`.

Если нужного файла документации ещё нет, Codex должен предложить создать его, но не менять структуру документации без утверждения пользователя.

---

## 3. Рабочий процесс

Codex не должен сразу писать код.

Текущая цель Фазы 1 — подготовить безопасный каркас проекта и документацию. До отдельного ревью и утверждения Фазы 1 запрещено "заодно" реализовывать рабочие продуктовые вертикали: калькулятор, авторизацию, админку, аналитику, обращения или публичный продуктовый интерфейс.

Перед реализацией каждой задачи нужно сначала показать:

1. краткий план;
2. список файлов, которые будут созданы или изменены;
3. нужны ли миграции;
4. какие тесты будут добавлены или обновлены;
5. есть ли риски;
6. есть ли вопросы, без которых нельзя безопасно продолжать.

После реализации Codex обязан показать:

1. что сделано;
2. список изменённых файлов;
3. как проверить результат;
4. какие тесты запустить;
5. какие документы обновлены;
6. какие допущения были сделаны;
7. были ли отклонения от ТЗ;
8. требуется ли решение пользователя.

## 4. Запреты MVP

В рамках MVP запрещено:

* реализовывать онлайн-оплату;
* реализовывать корзину;
* реализовывать личный кабинет клиента;
* реализовывать регистрацию клиентов;
* реализовывать wishlist;
* превращать проект в маркетплейс;
* добавлять микросервисы;
* добавлять Kafka / RabbitMQ без отдельного утверждения;
* добавлять платные обязательные сервисы;
* делать зависимость проекта от платных SaaS;
* раскрывать ASP.NET backend API напрямую наружу в production;
* переносить бизнес-логику на frontend;
* считать итоговую цену на frontend как источник истины;
* хранить секреты в коде;
* хранить `.env` в репозитории;
* отключать проверки безопасности ради удобства.

В рамках Фазы 1 дополнительно запрещено реализовывать:

* рабочий калькулятор;
* авторизацию;
* админ-панель;
* аналитику;
* обработку обращений;
* полноценный публичный продуктовый frontend.

Фаза 1 может содержать только документацию, безопасный скелет, конфигурационный каркас и пустые/минимальные placeholders, если они явно нужны для структуры проекта.

## 5. Архитектурные правила

Обязательный стек:

* .NET 10;
* ASP.NET Core 10;
* PostgreSQL;
* EF Core;
* CQRS + MediatR;
* AutoMapper;
* Nuxt 3;
* Nuxt BFF;
* Docker Compose;
* nginx reverse proxy.

Правила архитектуры:

* backend является source of truth для цены, прав доступа и бизнес-логики;
* frontend только отображает данные и собирает ввод пользователя;
* публичный браузер не должен напрямую обращаться к внутреннему ASP.NET API в production;
* frontend работает через Nuxt BFF;
* ASP.NET API доступен внутри docker-сети;
* `/admin/*` защищён авторизацией;
* Swagger / Scalar доступен только в development;
* production должен скрывать Swagger / Scalar и внутренние API-маршруты;
* бизнес-логика не должна находиться в контроллерах;
* доменная логика не должна зависеть от Infrastructure;
* Application layer не должен зависеть от API;
* Infrastructure реализует интерфейсы Application layer.

## 6. Критичные решения

Codex не имеет права самостоятельно принимать критичные решения.

Критичные решения:

* изменение технологического стека;
* замена .NET 10, Nuxt 3, PostgreSQL, AutoMapper, MediatR;
* изменение архитектуры backend/frontend;
* изменение структуры папок;
* изменение схемы БД;
* изменение доменной модели;
* изменение публичных API routes;
* изменение формата API response;
* изменение схемы авторизации;
* изменение reverse proxy схемы;
* добавление новой инфраструктурной зависимости;
* добавление платного сервиса;
* добавление личного кабинета клиента;
* добавление онлайн-оплаты;
* добавление корзины;
* изменение принципа “backend is source of truth”;
* удаление тестов;
* отключение security middleware;
* отключение rate limiting;
* хранение токенов менее безопасным способом.

Для критичного решения Codex обязан оформить предложение:

```md
## Предложение изменения

Что предлагается изменить:
Почему это нужно:
Какие есть альтернативы:
Плюсы:
Минусы:
Риски:
Какие файлы будут затронуты:
Какие документы нужно обновить:
Рекомендация Codex:
Требуется утверждение пользователя: да
```

После этого Codex должен остановиться и ждать утверждения пользователя.

## 7. Новые зависимости

Нельзя добавлять NuGet/npm/Docker-зависимости без причины.

Перед добавлением зависимости Codex обязан указать:

* название зависимости;
* зачем она нужна;
* бесплатная ли она;
* есть ли open-source лицензия;
* можно ли обойтись без неё;
* как она влияет на память/CPU;
* используется ли она в production;
* есть ли альтернатива проще.

Если зависимость критичная или инфраструктурная — требуется утверждение пользователя.

---

## 8. Безопасность

Обязательные правила:

* все входные данные валидируются;
* все команды/запросы проходят через FluentValidation, если применимо;
* пароли хранятся только в виде хеша;
* refresh token хранится в БД только в виде хеша;
* access token не хранится в localStorage;
* секреты только через env variables;
* `.env` должен быть в `.gitignore`;
* production не показывает stack trace;
* production не раскрывает Swagger / Scalar;
* CORS ограничен;
* rate limiting включён для публичных форм и auth endpoints;
* загружаемые файлы проверяются по размеру, MIME type и magic bytes;
* нельзя логировать пароли, токены и чувствительные персональные данные.

---

## 9. Производительность и минимальный VPS

Проект должен быть рассчитан на минимальную VPS-конфигурацию.

Правила:

* не добавлять тяжёлые зависимости без необходимости;
* не делать запросы без пагинации;
* не грузить все записи из БД без LIMIT;
* использовать индексы для frequently queried fields;
* изображения оптимизировать;
* использовать lazy loading;
* Redis/Valkey использовать только там, где есть смысл;
* MinIO использовать рационально;
* логи в production должны ротироваться;
* не включать debug/verbose logging в production.

## 10. Тесты

Новая бизнес-логика должна сопровождаться тестами.

Обязательно тестировать:

* расчёт цены;
* скидки;
* некорректные параметры калькулятора;
* создание обращения;
* смену статуса обращения;
* авторизацию админа;
* security-sensitive логику;
* маппинг AutoMapper, если есть риск ошибки.

Нельзя удалять или отключать тесты без утверждения пользователя.

Если тесты не проходят, Codex должен явно указать это и объяснить причину.

---

## 11. Definition of Done

Задача считается завершённой только если:

* код собирается;
* релевантные тесты добавлены или обновлены;
* релевантные тесты проходят или явно указано, почему они не запускались;
* документация обновлена;
* CHANGELOG обновлён;
* нет неутверждённых критичных решений;
* нет новых платных обязательных сервисов;
* нет переноса бизнес-логики на frontend;
* нет секретов в коде;
* пользователь получил список изменённых файлов;
* пользователь получил команды для проверки результата.

## 13. Главное правило

Проект создаётся не ради технологий.

Главный сценарий MVP:

```text
Клиент увидел работы → понял ценность → рассчитал примерную цену → написал мастеру.
```

Всё, что мешает этому сценарию, не должно попадать в MVP.
